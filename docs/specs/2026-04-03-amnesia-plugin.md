# Spec: Amnesia Claude Code Plugin

## Problem Statement

The current Amnesia system is implemented as natural-language instructions (`.md` files) that Claude Code interprets on every run. This makes memory operations fragile, slow, and non-portable — the files live inside a specific project and must be copied manually to other workspaces.

## Goals

- Deterministic logic (merge, dedup, validation) in Python code — no variation between runs
- Distributable as a native Claude Code plugin (`/plugin install`)
- `.md` commands with ≤40 lines — orchestration only, no embedded logic
- Compatible with all 17 existing JSON files (`amnesia-entity` schema)
- Keeps `progress.md` intact (outside plugin scope)

## Context

There are currently 4 command files (`amnesia.md`, `entity.md`, `project.md`, `progress.md`) with about 800 lines of instructions in total. They manage 17 entities with JSON memory in `.claude/amnesia/memory/` and session logs in `.claude/amnesia/sessions/`. The merge, deduplication, and schema validation logic is written in natural language — executed by LLM reasoning, not by code.

## Requirements

### P1 (MVP)

1. **[REQ-001]** WHEN the user invokes `/amnesia <name>` and `<name>` exists in `.claude/amnesia/memory/` THEN the system MUST call `amnesia.py entity load <name>` and Claude MUST present a summary in pt-BR.

2. **[REQ-002]** WHEN the user invokes `/amnesia <name> save` THEN the system MUST: (1) Claude extracts session facts in English, (2) call `amnesia.py entity diff <name> <updates_json>` to get a merge preview, (3) present the diff in pt-BR, (4) after confirmation call `amnesia.py entity save <name> <updates_json>`.

3. **[REQ-003]** WHEN `amnesia.py entity save` is called THEN the system MUST apply deterministic merge: `permanent_facts.items` and `technical_notes` with string-normalization dedup; `decisions` with dedup by `(date, author, decision[:50])`; `current_status` and `last_session` replaced entirely.

4. **[REQ-004]** WHEN the user invokes `/amnesia list` THEN the system MUST list all entities in `.claude/amnesia/memory/*.json` with `entity`, `updated_at`, and `current_status.phase` translated to pt-BR.

5. **[REQ-005]** WHEN the user invokes `/amnesia <name>` and `<name>` matches a project in `projects/` THEN the system MUST call `amnesia.py project load <name>` returning the latest sessions from the last 7 days plus the project's git log.

6. **[REQ-006]** WHEN the user invokes `/amnesia save` THEN Claude MUST detect session context and call the corresponding save flows for detected entities and projects.

7. **[REQ-007]** WHEN `amnesia.py` finds a file with schema `etl-client-memory-v1` THEN it MUST migrate it automatically to `amnesia-entity` on the next write, without data loss.

8. **[REQ-008]** WHEN the plugin is installed via `/plugin install` THEN the commands `amnesia.md`, `amnesia/entity.md`, and `amnesia/project.md` MUST become available in Claude Code for the target project.

### P2 (Should Have)

1. **[REQ-017]** WHEN an amnesia command produces conversation output for the user THEN it MUST respect the `language` field from Claude Code's `settings.json` (global `~/.claude/settings.json` or local `.claude/settings.json`) — if `language` is `"pt-BR"`, respond in Portuguese; if `"en"` or absent, respond in English. The commands `.md` MUST read this setting at runtime and adapt all user-facing text accordingly.

2. **[REQ-018]** WHEN any final document is generated by the plugin (README.md, CONTRIBUTING.md, session logs in `sessions/`, entity memory JSON values) THEN it MUST be written in English regardless of the `language` setting. User-facing output in conversation is language-aware; stored documents are always English.

3. **[REQ-009]** WHEN `amnesia.py entity diff` is called THEN it MUST return structured JSON with `added`, `updated`, and `skipped` fields per section.

4. **[REQ-010]** WHEN `amnesia.py project save` is called THEN it MUST append to `sessions/YYYY-MM-DD.md` with header `## project:<name> (HH:MM)` and create the file if it does not exist.

5. **[REQ-011]** WHEN `amnesia.py` receives invalid or corrupted JSON THEN it MUST return `{"error": "...", "file": "..."}` instead of raising an exception.

6. **[REQ-014]** WHEN implementation is complete THEN the project MUST include `README.md` in English with: overview, prerequisites, installation via `/plugin install`, usage for each command, file structure, and output examples.

7. **[REQ-015]** WHEN implementation is complete THEN the project MUST include `LICENSE` with the MIT license.

8. **[REQ-016]** WHEN implementation is complete THEN the project MUST include `CONTRIBUTING.md` in English with: development setup, how to run tests, code conventions, PR submission process, and the `amnesia.py` structure.

### P3 (Nice to Have)

1. **[REQ-012]** WHEN the user invokes `/amnesia list` THEN the system MUST list entities and projects in a unified view.

2. **[REQ-013]** WHEN `amnesia.py` is executed with `--project-root <path>` THEN it MUST use that path instead of detecting it via `git rev-parse`.

## Edge Cases

- **Entity/project conflict** — prioritize entity; use `/amnesia entity <name>` or `/amnesia project <name>` to disambiguate
- **Corrupted JSON** — return `{"error": "invalid_json", "file": "..."}`, command presents an error and suggests recreation
- **Legacy schema (`etl-client-memory-v1`)** — read transparently, migrated on next write
- **Empty session** — report "No activity detected in this session"
- **Duplicates** — dedup by normalization (strip + lowercase for comparison, preserve original)
- **Missing `memory/` directory** — created automatically
- **`git rev-parse` fails** — fallback to `$PWD`

## Architecture Decisions

1. **Plugin as commands + Python CLI (not MCP)** — simplicity over protocol. `amnesia.py` is a CLI invoked via the Bash tool. Distributed as a native Claude Code plugin (`.claude-plugin/plugin.json`).
2. **Separation of responsibilities** — `amnesia.py` handles deterministic behavior (I/O, merge, dedup, Pydantic validation); Claude handles semantic behavior (fact extraction, translation, presentation); `.md` commands are thin orchestrators.
3. **Two-step save** — `entity diff` returns a preview → user confirms → `entity save` writes. Prevents accidental overwrites in permanent memory.
4. **Structured JSON output** — `amnesia.py` always returns JSON (`{"data": ..., "error": null}`), never plain text. Claude formats it for the user.
5. **`progress.md` out of scope** — remains a separate `.md` command and depends on external sources (multi-git, Notion, session analyzer).

## Constraints

- Python 3.10+, with Pydantic v2 as the only external dependency
- `amnesia.py` runs without a virtualenv — installed via `uv tool install` or `pip install --user`
- All operations are local, with no network dependency
- Compatible with Claude Code's `.claude-plugin/plugin.json` format

## Out of Scope

| Item | Reason |
|------|-------|
| `progress.md` / session snapshot | Depends on external sources (multi-git, Notion, session analyzer) |
| Notion synchronization | Responsibility of an external command |
| Automatic translation via API | Claude handles it semantically |
| Old session compaction (90 days) | Not justified for the MVP |
| Support for multiple simultaneous workspaces | `--project-root` solves it per invocation |

## Test Strategy

- **Unit:** `amnesia.py` directly — merge logic, dedup, schema migration, edge cases (corrupted JSON, missing directory, legacy schema)
- **Integration:** Fixtures with real JSON files from the 17 clients — validate load/diff/save roundtrip without data loss
- **Manual:** Install the plugin in a test project, run `/amnesia my-project`, `/amnesia save`, `/amnesia list`, and validate output
